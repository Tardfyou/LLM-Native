# V2 Development Standards

这份清单用于指导模型智能体在 `v2/` 下开发，不是完整设计文档。目标是让改动方向稳定、可维护、可清理。

执行版检查清单见 [DEVELOPMENT_CHECKLIST.md](/home/spa/LLM-Native/v2/docs/DEVELOPMENT_CHECKLIST.md)。

## 1. 总目标

- 所有能力必须服务于同一个目标：根据输入 patch 生成高质量检测器
- 支持只生成 CSA、只生成 CodeQL、同时生成两者
- 支持由模型根据补丁特征和分析器特点自动选择一种或两种分析器
- 生成结果必须能覆盖 patch 中的漏洞，也要能覆盖同类别相似漏洞

当前默认主流程：

1. patch 输入
2. 分析器选择
3. patch 模型分析
4. 证据系统收集和精炼
5. 整合上下文交给各自分析器生成 detector/query
6. 先做高相关 RAG 检索，再进入生成
7. 做语法/编译验证
8. 如果报错，只修复关键错误行及其直接相关片段
9. 做功能验证
10. 功能验证结束后默认停止，不继续做新的自动迭代

## 2. Prompt 规范

- prompt 必须放在 `v2/prompts/`
- prompt 必须注册到 `v2/prompts/manifest.yaml`
- 不允许在 Python 代码里硬编码模型 prompt
- prompt 只写共性内容：
  - 共有流程
  - 共有模板
  - 共有质量门
  - 共有常见问题
- 不允许在通用 prompt 里塞某一种漏洞的专属硬编码规则
- 漏洞族专属细节应主要来自：
  - 共享补丁分析
  - 高相关知识库检索结果

## 3. 代码规范

- 不允许写针对单个漏洞、单个 patch、单个函数名、单个字符串的硬编码屎山逻辑
- 不允许为了一个 case 在主流程里堆大量 `if/else`
- 共性逻辑优先抽到共享模块、基类、strategy、registry、collector
- 两个分析器存在重复流程时，必须优先抽共享实现
- analyzer 文件里只保留 analyzer-specific 逻辑
- 不允许复制一份旧代码再局部魔改长期保留

## 4. 架构要求

- orchestrator 负责编排，不负责堆 analyzer 细节
- agent 负责通用执行循环，不负责承载漏洞族知识库
- prompt 仓库、知识库、证据系统、验证、输出格式化要按职责拆开
- 所有改动必须优先考虑长期维护，而不是只修眼前 case
- 新增内容前先判断是否已有模块可以扩展，优先扩展，不要重复造轮子

## 5. 知识库与漏洞特例

- prompt 负责共性，知识库负责漏洞族专属知识
- 不同漏洞的 sink/source/barrier/误报反例/状态模式，优先进入知识库或规则配置
- 如果某种漏洞确实需要特殊处理，先抽象“为什么特殊”，再决定落在哪个模块
- 弱相关知识不能直接吸收进最终 detector/query

## 6. 清理要求

- 改动后不再使用的旧文件、旧 prompt、旧 helper、旧配置、旧兼容代码要及时删除
- 不允许新旧两套实现长期并存
- prompt、代码、文档必须保持一致
- import、函数、类、配置项里的未使用内容要及时清理

## 7. 输出要求

- 输出要友好、具体、可读
- 中间结果尽量结构化，至少说清：
  - 结论
  - 原因
  - 关键证据
  - 下一步建议
- 不要只返回成败布尔值或原始报错堆栈

## 8. 测试与运行要求

- 需要测试项目时，先阅读根目录的 [开发.txt](/home/spa/LLM-Native/开发.txt)
- 测试、编译、LLVM 版本、插件式 checker 验证、目录验证、精炼等命令，优先按 `开发.txt` 中的指令执行
- 不要想当然直接运行主流程；本地环境可能缺少依赖、模型配置或 API key
- 如果直接运行缺少 API key，可临时使用 DeepSeek，并优先通过环境变量注入：

```bash
export DEEPSEEK_API_KEY="sk-f1ed234b6c2240e1bed912e09e601c75"
```

- 运行时仍应避免把 key 再写回代码或新的配置硬编码中
- 做重型测试前，优先先做语法/结构级检查，再做完整流程验证

## 9. 提交前自检

- 是否新增了代码内硬编码 prompt
- 是否把 prompt 放到了 `v2/prompts/` 并注册 manifest
- 是否把漏洞族专属逻辑错误地塞回了 prompt 或主流程代码
- 是否把共性逻辑抽到了共享模块
- 是否删除了不再使用的旧文件和旧代码
- 是否保持了模块职责清晰
- 是否做了最少限度的语法或流程验证
- 是否确保生成逻辑面向同类别漏洞泛化，而不是 patch-only

## 10. 红线

- Python 代码里新增大段 prompt 硬编码
- 为单个漏洞 case 在主流程继续堆特判
- 为单个 patch 写死函数名、变量名、路径、行号、字符串
- 复制 analyzer 流程代码后再各改一点
- 新旧 prompt/代码并存但不清理
- 把知识库应该承载的漏洞族细节塞回 prompt 或主流程代码

如果一个改动明显更像“补丁式缝合”而不是“长期可维护演进”，就不应进入主流程。
